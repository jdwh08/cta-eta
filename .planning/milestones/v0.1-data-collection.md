# Milestone v0.1: Data Collection

**Status:** ✅ SHIPPED 2026-02-16
**Phases:** 1-9
**Total Plans:** 22

## Overview

Production-grade data collection pipeline for CTA train positions and weather data, ready for months of continuous operation. The system runs continuously, capturing ~230k train position snapshots daily across all 8 CTA lines with 15-second polling, hourly weather data from multi-source APIs, and full operational infrastructure (monitoring, alerting, systemd deployment).

### Phase 7: Resilience & Recovery

**Goal**: Intelligent retry logic, gap detection, and graceful shutdown with state preservation
**Depends on**: Phase 6
**Plans**: 3 plans

Plans:

- [x] 07-02: Gap detection with TDD and Parquet metadata
- [x] 07-03: Graceful shutdown with state preservation and restart recovery

**Details:**

## Milestone Summary

**Key Decisions:**

- Use fsspec for unified S3/GCS storage API (cloud-agnostic from day one)
- 3 AM Chicago timezone as daily partition boundary (avoids midnight-UTC confusion)
- Lazy weather grid discovery avoids 300+ API calls on startup
- aiometer rate limiting across all external API calls (consistent, diagnostic, future-proof)
- OpenWeatherMap as fallback-only to conserve very limited free tier quota
- `dataset_name` parameter for organized multi-dataset Parquet storage
- CLI monitoring over web dashboard (simpler, scriptable, no server process needed)
- Email alerting over webhook/Slack (no external service dependency for critical alerts)
- `copytruncate` log rotation for open JSONL file handles (no service restart needed)

**Issues Resolved:**

- Single-train XML-to-JSON dict normalization bug in CTA API client
- `datetime.UTC` import compatibility issue
- Phase 8 monitoring server prototyped then replaced with CLI approach (08-02-SUMMARY.md and server.py deleted)

**Issues Deferred:**

- Cloud VM deployment and actual production operation (requires hardware provisioning)
- OpenWeatherMap integration testing (limited by free tier quota)
- Model training infrastructure (Phase 2 — after months of data collection)

**Technical Debt Incurred:**

- No integration tests for full daemon-to-storage pipeline (only unit tests and manual integration run)
- Weather daemon lacks the same level of integration validation as train daemon (Phase 6 did 5-minute live test)
- OpenWeatherMap fallback path minimally tested due to API quota constraints
