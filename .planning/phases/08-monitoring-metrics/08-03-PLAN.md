  <action>
Create `health.py` module with health assessment logic:

**Function: `assess_daemon_health(daemon_name: str) -> dict[str, object]`**
Aggregate health data for a single daemon:
1. Read daemon state from `.daemon_state/{daemon_name}.json`
2. Read latest metrics from `.daemon_state/{daemon_name}.metrics.jsonl` (last line)
3. Parse metrics for success_rate, error_rate, total_calls
4. Check staleness: last_poll_timestamp > 5 minutes ago = stale
5. Count gaps from last 24h metrics window
6. Return: {
     daemon: str,
     status: "healthy" | "degraded" | "critical" | "unknown",
     last_poll_timestamp: float | None,
     is_stale: bool,
     metrics: {success_rate, error_rate, total_calls_24h},
     gap_count_24h: int,
     issues: [str]  # Human-readable issue descriptions
   }

**Status classification logic (simplified, refined in Task 2):**
- critical: daemon stale (>5 min) OR success_rate < 0.5 OR gap_count > 50
- degraded: success_rate < 0.9 OR error_rate > 0.1 OR gap_count > 10
- healthy: all checks pass
- unknown: no data (daemon never run)

**Function: `assess_overall_health() -> dict[str, object]`**
Aggregate health across all daemons:
1. Discover all daemon state files in `.daemon_state/`
2. Call assess_daemon_health() for each
3. Return: {
     overall_status: "healthy" | "degraded" | "critical",
     daemons: {daemon_name: daemon_health},
     summary: {total_daemons, healthy_count, degraded_count, critical_count},
     timestamp: float
   }

**Overall status logic:**
- critical: any daemon critical
- degraded: any daemon degraded (and none critical)
- healthy: all daemons healthy

Follow existing patterns: Use Path for file operations, handle missing files gracefully (return unknown status), use structured logging for health checks, complete type hints, Google-style docstrings.

AVOID raising exceptions on missing files (daemons may not have run yet). AVOID reading entire metrics.jsonl into memory (use tail or read last line only).

<task type="auto">
  <name>Task 2: Define configurable alert thresholds for Phase 9</name>
  <files>src/cta_eta/monitoring/health.py, config.toml</files>
  <action>
Add configurable thresholds to config.toml and refine health assessment:

**Add config.toml section:**
```toml
[alert_thresholds]
# Staleness thresholds
daemon_stale_minutes = 5.0  # Daemon hasn't polled in this many minutes

# Success rate thresholds (0.0 to 1.0)
success_rate_degraded = 0.90  # Below this = degraded
success_rate_critical = 0.50  # Below this = critical

# Error rate thresholds (0.0 to 1.0)
error_rate_degraded = 0.10   # Above this = degraded
error_rate_critical = 0.30   # Above this = critical

# Gap thresholds (count in last 24h)
gap_count_degraded = 10      # More than this = degraded
gap_count_critical = 50      # More than this = critical

# Minimum calls for rate calculations
min_calls_for_rates = 10     # Ignore rates if fewer calls (avoid false alarms)
```

**Update health.py:**
1. Create `AlertThresholds` dataclass to hold loaded config
2. Load thresholds via `get_config_section("alert_thresholds")` with defaults
3. Pass thresholds to assess_daemon_health() and apply to status logic
4. Return threshold values in health response for Phase 9 context
5. Add method `get_alert_context(daemon_health: dict) -> dict` that returns:
   - Which thresholds were violated
   - Suggested alert message text
   - Alert severity level (warning vs critical)
   - Should Phase 9 send email notification?

**Status classification logic (refined):**
Apply thresholds in order of severity:
- critical: is_stale OR success_rate < critical threshold OR gap_count > critical threshold
- degraded: success_rate < degraded threshold OR error_rate > degraded threshold OR gap_count > degraded threshold
- healthy: all thresholds pass AND min_calls_for_rates met
- unknown: no data OR too few calls (< min_calls_for_rates)

Follow existing patterns: Config validation on load (log warning for invalid values), use dataclass for threshold config, document each threshold in code comments.

AVOID hardcoded thresholds (all must be configurable). AVOID alerting on insufficient data (check min_calls_for_rates first).
