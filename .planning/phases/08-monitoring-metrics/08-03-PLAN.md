**Command: `cta-monitor status`**
Quick health check showing daemon health, last collection times, overall state.

Implementation:
1. Discover all daemon state files in `.daemon_state/*.json` (exclude .diagnostics.json, .events.jsonl, .metrics.jsonl)
2. For each daemon state file:
   - Read JSON: last_poll_timestamp (or last_collection_timestamp), total_records_collected, interval config
   - Calculate staleness: current_time - last_poll_timestamp
   - Classify: stale (>5 min), active (<5 min), unknown (missing timestamp)
3. Display table format:
   ```
   CTA Data Collection Status
   ==========================

   Daemon              Status    Last Poll        Records    Staleness
   ──────────────────────────────────────────────────────────────────
   TrainPositionDaemon  active   2m 15s ago       1,038      -
   WeatherDaemon        stale    2h 31m ago       145        STALE

   Overall: DEGRADED (1 stale daemon)
   ```
4. Exit code: 0 if all active, 1 if any stale, 2 if all unknown

**Command: `cta-monitor errors [--limit N] [--json]`**
Recent failures, API errors, what's broken and why.

Implementation:
1. Read diagnostic events from `.daemon_state/*.diagnostics.json` or `.daemon_state/*.events.jsonl`
2. Filter for error events (span_name ending in ".error" or event_type="error")
3. Extract: timestamp, daemon_name, span_name, error_type, error_message
4. Sort by timestamp descending (most recent first)
5. Apply --limit (default: 20)
6. Display table format:
   ```
   Recent Errors (last 20)
   =======================

   Time             Daemon              Error Type        Message
   ───────────────────────────────────────────────────────────────────
   2m ago          TrainPositionDaemon  httpx.TimeoutError  Request timeout after 10.0s
   15m ago         WeatherDaemon        httpx.HTTPStatusError  429 Rate limit exceeded
   ```
7. If --json flag: output as JSON array for machine consumption

**Shared utilities:**
- `_discover_daemons() -> list[str]`: scan .daemon_state/ for daemon names
- `_read_daemon_state(daemon_name: str) -> dict | None`: read daemon state JSON
- `_format_duration(seconds: float) -> str`: "2m 15s", "2h 31m", "3d 5h"
- `_classify_status(staleness: float) -> str`: "active", "stale", "unknown"

**CLI design patterns:**
- Use argparse (built-in, no extra dependencies)
- Follow Unix conventions: 0=success, 1=degraded, 2=critical/unknown
- Table output uses simple ASCII formatting (no dependencies like tabulate)
- JSON output via --json flag (for Phase 9 automation)
- Handle missing files gracefully (log warning, continue)
- Complete type hints, Google-style docstrings

AVOID using external CLI frameworks (click, typer - keep lightweight). AVOID raising exceptions on missing files (report as unknown status). AVOID complex formatting libraries (keep VPS resource usage minimal).

**Command: `cta-monitor gaps [--dataset NAME] [--days N] [--json]`**
Data collection gaps from Parquet file metadata.

Implementation:
1. Scan Parquet files in data directory (default: `./data/train_positions/`)
2. Option --dataset to specify dataset name (default: "train_positions")
3. Option --days to limit time window (default: 7 days)
4. For each Parquet file in time window:
   - Read schema metadata using pyarrow: `pq.read_metadata(file).schema.metadata`
   - Decode metadata bytes: `json.loads(metadata[b'gap_metadata'].decode())`
   - Extract gap info: is_gap, gap_duration_seconds, gap_reason, missed_poll_cycles
5. Aggregate gaps: count, total duration, reasons breakdown
6. Display table format:
   ```
   Data Collection Gaps (last 7 days)
   ===================================

   Date          Duration   Reason            Missed Cycles
   ──────────────────────────────────────────────────────────
   2026-01-27    2m 35s     retry_exhausted   10
   2026-01-26    1h 15m     downtime          300

   Summary: 2 gaps, 1h 17m total, 310 cycles missed
   ```
7. If --json flag: output gap list as JSON array

**Command: `cta-monitor metrics [--window HOURS] [--json]`**
Aggregated metrics for Phase 9 alerting.

Implementation:
1. Read latest metrics from `.daemon_state/*.metrics.jsonl` (last line of each file)
2. Parse JSONL: timestamp, daemon_name, metrics_1h (or metrics_24h)
3. Extract: success_rate, error_rate, total_calls, latency_percentiles
4. Calculate overall health:
   - Critical: any daemon success_rate < 0.5
   - Degraded: any daemon success_rate < 0.9
   - Healthy: all daemons success_rate >= 0.9
5. Display table format:
   ```
   Metrics Summary (1h window)

   WeatherDaemon        100.0%       0.0%        2      1.2s

   Overall Health: HEALTHY
   ```
