<objective>
Implement intelligent retry logic with CTA-specific error code handling and extended backoff windows.

Purpose: Maximize data collection within API limits by distinguishing between "keep trying" errors (network timeouts) and "stop trying" errors (daily quota, invalid parameters). CTA's Train Tracker API has no historical data endpoint - whatever we collect during each poll is the only source of truth, making aggressive retry critical.

Output: Enhanced error classification supporting CTA error codes (102, 100, 101, 106, 107, 500), extended retry windows (5-10 minutes for network errors, hours for daily quota), and poll blocking during retry to prevent cascading failures.
</objective>

<task type="auto">
  <name>Task 2: Implement extended retry with poll blocking</name>
  <files>src/cta_eta/data_collection/orchestration/train_position_daemon.py</files>
  <action>
Enhance TrainPositionDaemon's run loop to handle CTA error categories with appropriate retry windows:

**Retry Strategy by Error Category:**
1. **ErrorCategory.TRANSIENT** (network timeouts, 5xx errors):
   - Retry the whole _collect_train_positions_cycle (wrap in stamina or an explicit retry loop) with extended max wait (5-10 minutes total)
   - Block subsequent polls during retry - await retry completion before next cycle
   - If retry exhausted, log gap, set pending_gap_metadata for Plan 07-02, and return to 15-second schedule

2. **ErrorCategory.DAILY_QUOTA** (CTA error 102):
   - Log critical warning: daily API limit hit
   - **Bounded probe (configurable):** probe_102_attempts (default 2) and probe_102_intervals (e.g. [300, 900] seconds = 5 min, 15 min), e.g. under [collection] in config.toml. If time until midnight Chicago is under 15 minutes, skip probe and go straight to midnight sleep.
   - For each probe: sleep the interval (using self.sleep or chunked asyncio.sleep with _shutdown/self.running checks so SIGTERM can complete), then one poll. On success, resume 15-second schedule. On 102 again, next probe or proceed to midnight sleep.
   - If all probes return 102: compute next midnight America/Chicago + 5 minutes buffer; sleep until that time in chunks with shutdown checks. Then resume 15-second schedule.
   - Don't queue missed polls. Accept the gap, flag it (Plan 07-02), and resume.

3. **ErrorCategory.CONFIGURATION** (CTA errors 100/101/106/107/500):
   - Log fatal error with specific error code and message
   - Exit daemon gracefully (these require manual intervention)

4. **ErrorCategory.RATE_LIMIT** (HTTP 429):
   - Apply existing 2x backoff pattern; block poll during backoff

**Implementation approach:**
- Wrap _collect_train_positions_cycle() in try/except; classify with daemon_utils.classify_error()
- Match on ErrorCategory and apply the above. For DAILY_QUOTA, all sleeps (probe intervals and midnight wait) must be chunked and shutdown-interruptible so that "shutdown during retry" completes in reasonable time.
- Track retry/probe attempts in diagnostic spans
- Preserve state: last_poll_timestamp updated only on successful store

**What to avoid and WHY:**
- Don't queue missed polls - accept the gap, flag it (Plan 07-02), return to schedule
- Don't use time.sleep() - use self.sleep or chunked await asyncio.sleep with _shutdown checks
- Don't retry CONFIGURATION - requires code/config changes
- Don't use UTC for quota reset - use ZoneInfo("America/Chicago")
- Don't "poll at lower frequency" until midnight for 102 - it doesn't help when 102 is real and delays recovery when it's a false positive. Use a bounded probe then sleep until midnight.
